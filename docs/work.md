# ToyC 单人完成流程

当前 `clone/` 里的代码只是一个里程碑 1 原型：它只支持 `int main() { return <一元常量表达式>; }`，还没有进入完整实验要求。一个人完成时，不要边写边扩接口，应该按下面顺序推进。

## 1. 先固定总目标

最终程序需要满足：

- 使用 `C++20 + CMake + Flex + Bison/Yacc`
- 保留 `ANTLR 4.13.1` 环境作为文法对照或备用前端原型
- 从 `stdin` 读取 ToyC 源程序
- 向 `stdout` 输出 RISC-V32 汇编
- 支持 `-opt`
- 覆盖题目要求的全部语法、语义和基础代码生成

## 2. 先做一次基线验证

每次大改之前先确认当前版本可构建、可运行：

```bash
cmake -S clone -B clone/build
cmake --build clone/build
./clone/build/compiler < clone/tests/milestone1/return_42.tc
./clone/build/compiler < clone/tests/milestone1/unary.tc
```

如果这一步不稳定，不要直接扩功能。

## 3. 推荐开发顺序

### 阶段 A：重构 AST 和总控骨架

目标：把“只能返回常量”的 AST 扩成完整编译器前端骨架。

要做的事：

- 扩展 `ast.hpp` / `ast.cpp`
- 增加源码位置 `SourceLocation`
- 增加声明、语句、表达式节点
- 把 `Program` 改成真正的 `CompUnit`
- 主程序拆成：
  - 解析
  - 语义分析
  - IR 生成
  - RISC-V32 代码生成

完成标准：

- 结构上可以表示全局变量、全局常量、函数、块、if、while、break、continue、return、赋值、函数调用

### 阶段 B：扩展词法和语法

目标：让 Flex/Bison 覆盖完整 ToyC 文法。

要做的事：

- 在 `lexer.l` 中补全关键字、运算符、注释、整数字面量、标识符
- 在 `parser.y` 中补全：
  - `CompUnit`
  - `Decl`
  - `ConstDecl`
  - `VarDecl`
  - `FuncDef`
  - `Block`
  - `Stmt`
  - `Expr`
  - 所有二元/一元表达式层级
- 用优先级和结合性消解表达式冲突
- 支持函数调用和参数列表

完成标准：

- 所有合法 ToyC 程序都能生成 AST
- 语法错误能输出可读信息

### 阶段 C：补全语义分析

目标：把题目要求里的语义约束全部检查完。

要做的事：

- 实现符号表和作用域栈
- 区分变量、常量、函数、形参
- 做先声明后使用检查
- 做重定义检查
- 检查 `main` 的签名
- 检查函数调用是否出现在被调函数定义之后，允许递归
- 检查 `break/continue` 的上下文
- 检查 `int` 函数所有路径返回
- 检查 `void` 函数返回语句合法
- 检查常量初始化表达式可编译期求值
- 检查常量不能作为赋值左值

完成标准：

- 对非法输入能拒绝编译
- 对合法输入能产出后端可消费的中间表示

### 阶段 D：定义统一 IR

目标：把前端和后端解耦，避免代码生成直接依赖 AST。

建议 IR 至少包含：

- 模块
- 全局对象
- 函数
- 基本块
- 三地址码风格指令

建议指令至少覆盖：

- 常量加载
- 全局/局部读写
- 一元/二元运算
- 比较
- 条件跳转
- 无条件跳转
- 函数调用
- 返回

完成标准：

- AST 经语义分析后可以稳定降到 IR

### 阶段 E：完成 RISC-V32 后端

目标：把 IR 翻译为能跑通测试的汇编。

要做的事：

- 生成 `.text` / `.data`
- 处理全局变量和全局常量
- 设计栈帧布局
- 处理局部变量分配
- 处理形参和返回值
- 处理条件分支和循环
- 实现短路逻辑
- 保证函数调用约定一致

完成标准：

- 生成的汇编能在 RISC-V32 环境下正确运行

### 阶段 F：补 `-opt` 和基础优化

目标：先保证正确性，再做能拿分的优化。

建议优先做：

- 常量折叠
- 常量传播
- 死代码删除
- 简单冗余跳转消除
- 简单寄存器/临时值复用

完成标准：

- `-opt` 打开后结果不变，性能更好或至少不退化

## 4. 每阶段的验证方式

建议按里程碑建立测试目录：

- `tests/milestone1/`：常量返回、一元表达式
- `tests/milestone2/`：二元表达式、优先级、括号
- `tests/milestone3/`：局部变量、赋值、作用域
- `tests/milestone4/`：if/else、while、break、continue
- `tests/milestone5/`：函数调用、递归、全局变量/常量
- `tests/milestone6/`：语义错误用例
- `tests/milestone7/`：优化与性能样例

每次改动至少做三类检查：

- 能否编译通过
- 对合法程序是否输出汇编
- 对非法程序是否给出错误

## 5. 当前版本的定位

当前版本只适合作为“最小闭环起点”，不适合直接在原结构上硬补全部特性。建议下一步优先做两件事：

1. 重构 AST，去掉当前只面向 `return <expr>` 的结构限制。
2. 把语义分析和代码生成从 `main.cpp` 中拆出去，形成明确阶段边界。

## 6. 实施原则

- 每次只扩一层能力，不要同时改语法、语义、后端三层大逻辑。
- 先让最小子集跑通，再扩下一类语句。
- 后端不要直接读 AST，必须经过统一 IR。
- 优化永远放在正确性之后。

如果要继续推进，下一步应先把 AST 扩成完整的 `CompUnit + Decl + FuncDef + Stmt + Expr` 结构，然后同步改造 `parser.y`。
