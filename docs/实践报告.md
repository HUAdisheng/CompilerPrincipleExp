# ToyC 编译器实践报告

## 一、实验背景与目标

本次实践的任务是实现一个 ToyC 语言编译器。ToyC 是 C 语言的一个简化子集，保留了整数、函数、变量、常量、表达式、条件分支和循环等核心结构，去除了数组、指针、输入输出和多文件编译等复杂特性。实验要求编译器从标准输入读取 ToyC 源程序，经过词法分析、语法分析、语义分析和代码生成等阶段，最终向标准输出生成可执行语义正确的 RISC-V32 汇编代码。

本项目采用 C++20 实现，使用 Flex 完成词法分析，使用 Bison 完成语法分析，使用 CMake 和 Make 提供构建入口。编译器主程序支持普通编译模式和 `-opt` 优化模式，整体目标是在保证功能正确性的基础上，为后续性能优化保留扩展空间。

实验主要目标如下：

1. 完整覆盖 ToyC 语言定义中的语法结构。
2. 构建能够表达完整程序结构的抽象语法树。
3. 实现符号表、作用域、类型和语义约束检查。
4. 生成符合基本调用约定的 RISC-V32 汇编。
5. 建立分阶段测试用例，验证合法程序和非法程序的处理结果。
6. 在 `-opt` 模式下实现基础常量相关优化。


编译器的整体流程如下：

```text
ToyC 源程序
    ↓
Flex 词法分析
    ↓
Bison 语法分析
    ↓
AST 构建
    ↓
语义分析
    ↓
RISC-V32 汇编生成
```

`main.cpp` 是编译器的总控入口。它首先解析命令行参数，识别是否启用 `-opt`；随后调用 `yyparse()` 读取标准输入并构建 AST；解析成功后调用 `analyzeProgram()` 进行语义分析；如果语义分析通过，则调用 `generateRiscV()` 输出 RISC-V32 汇编。语义错误会输出到标准错误，并以非零状态码退出。

本项目目前采用 AST 驱动的直接代码生成方式，即语义分析之后直接遍历 AST 生成汇编。该方案实现路径较短，适合课程实践中快速形成完整闭环；不足是中端优化空间受限，后续如果继续提升性能，可以引入三地址码 IR 和控制流图。

## 三、词法分析设计与实现

词法分析由 `src/lexer.l` 实现，使用 Flex 将 ToyC 源码切分为 Bison 可识别的 token。

### 3.1 关键字与标识符

ToyC 支持如下关键字：

```text
const int void if else while break continue return
```

词法分析器先匹配关键字，再匹配普通标识符。标识符规则与 C 语言基本一致：

```text
[_A-Za-z][_A-Za-z0-9]*
```

匹配到标识符时，词法分析器会将其保存为 `std::string`，通过 `yylval.text` 传递给语法分析器。

### 3.2 整数字面量

整数常量使用十进制形式，规则为：

```text
0|[1-9][0-9]*
```

词法分析阶段使用 `std::stoll` 将文本转换为整数。如果转换失败，会输出整数超范围错误。语法分析阶段再将该值转换为 32 位整数语义下的值。

需要说明的是，ToyC 文法中负数可以通过一元负号表达，例如 `-1` 会被解析为一元表达式，而不是在词法层直接作为带符号整数处理。这种做法更贴近常见编译器前端设计，也便于统一处理表达式优先级。

### 3.3 运算符、分隔符与注释

词法分析器支持：

- 算术运算符：`+`、`-`、`*`、`/`、`%`
- 关系运算符：`<`、`>`、`<=`、`>=`、`==`、`!=`
- 逻辑运算符：`&&`、`||`、`!`
- 赋值与分隔符：`=`、`;`、`,`、`(`、`)`、`{`、`}`

空白字符会被直接跳过。注释支持 C 风格的单行注释和多行注释：

```c
// line comment
/* block comment */
```

如果遇到无法识别的字符，词法分析器会输出错误并返回 `YYUNDEF`，使编译过程失败。

## 四、语法分析与 AST 构建

语法分析由 `src/parser.y` 实现，使用 Bison 根据 ToyC 文法构造 AST。语法分析的开始符号为 `compilation_unit`，对应整个编译单元。

### 4.1 顶层结构

ToyC 顶层结构由若干全局声明和函数定义组成。当前实现支持：

- 全局常量声明：`const int a = expr;`
- 全局变量声明：`int a = expr;`
- `int` 函数定义
- `void` 函数定义

语法分析器将顶层结构转换为 `TopLevelItem` 的派生节点：

- `GlobalDecl`
- `FuncDef`

整个程序由 `CompUnit` 节点保存。

### 4.2 语句结构

语句节点统一继承自 `Stmt`。当前 AST 支持如下语句：

- `BlockStmt`：语句块
- `EmptyStmt`：空语句
- `ExprStmt`：表达式语句
- `AssignStmt`：赋值语句
- `DeclStmt`：局部声明语句
- `IfStmt`：条件语句
- `WhileStmt`：循环语句
- `BreakStmt`：`break`
- `ContinueStmt`：`continue`
- `ReturnStmt`：返回语句

Bison 中使用如下规则解决悬空 `else` 问题：

```bison
%nonassoc LOWER_THAN_ELSE
%nonassoc ELSE
```

不带 `else` 的 `if` 规则使用 `%prec LOWER_THAN_ELSE`，使 `else` 优先结合最近的未匹配 `if`。

### 4.3 表达式结构

表达式节点统一继承自 `Expr`。当前 AST 支持：

- `IntLiteral`：整数常量
- `IdentifierExpr`：变量或常量引用
- `UnaryExpr`：一元表达式
- `BinaryExpr`：二元表达式
- `CallExpr`：函数调用表达式

表达式优先级通过语法非终结符分层实现，而不是依赖大量优先级声明。分层顺序如下：

```text
expression
logical_or_expression
logical_and_expression
relational_expression
additive_expression
multiplicative_expression
unary_expression
primary_expression
```

这种设计的优点是结构清晰，AST 天然反映表达式的优先级和结合性。例如 `1 + 2 * 3` 会被解析为 `1 + (2 * 3)`，而不是 `(1 + 2) * 3`。

### 4.4 内存管理

AST 节点在语法动作中临时使用裸指针创建，进入 AST 构造函数后统一转移为 `std::unique_ptr` 管理。例如 `BlockStmt` 会接收 `std::vector<Stmt*>`，然后逐个转为 `std::unique_ptr<Stmt>`。这种方式兼顾了 Bison 语义值传递的便利性和 C++ 侧的资源自动释放。

## 五、抽象语法树设计

AST 定义在 `src/ast.hpp` 和 `src/ast.cpp` 中。其核心设计是将 ToyC 程序划分为顶层项目、声明、语句和表达式四类对象。

### 5.1 基础节点

所有节点都继承自 `Node`：

```cpp
class Node {
public:
    virtual ~Node() = default;
    SourceLocation location{};
};
```

`SourceLocation` 为后续错误定位预留了行列信息。当前实现已经具备该结构，但语法构造阶段尚未充分填充位置，因此错误信息主要以语义消息为主，行列信息还有继续完善空间。

### 5.2 类型系统

ToyC 只支持 `int` 和 `void` 两类返回类型，因此使用枚举表示：

```cpp
enum class ValueType {
    Int,
    Void,
};
```

表达式语义上只允许产生 `int` 值；`void` 主要出现在函数返回类型和函数调用结果检查中。

### 5.3 顶层与函数节点

`CompUnit` 保存所有顶层项目。`FuncDef` 保存函数返回类型、函数名、形参列表和函数体：

```cpp
class FuncDef final : public TopLevelItem {
public:
    ValueType returnType;
    std::string name;
    std::vector<std::string> params;
    std::unique_ptr<BlockStmt> body;
};
```

全局变量和全局常量统一使用 `Decl` 表示，再由 `GlobalDecl` 包装为顶层节点。局部声明也复用同一个 `Decl`，由 `DeclStmt` 包装为语句节点。这减少了声明逻辑的重复。

## 六、语义分析设计与实现

语义分析由 `src/semantic.cpp` 实现，入口函数为：

```cpp
SemanticResult analyzeProgram(const CompUnit& program, const CompileOptions& options);
```

语义分析结果包含：

- 是否成功：`ok`
- 诊断信息：`diagnostics`
- 后端所需的程序信息：`ProgramInfo`

`ProgramInfo` 保存全局常量、全局变量、函数签名、函数定义顺序和全局变量顺序，供代码生成阶段使用。

### 6.1 符号表与作用域

语义分析器维护一个作用域栈：

```cpp
std::vector<Scope> scopes_;
```

每个 `Scope` 保存当前块内声明的变量、常量和形参。查找变量时，从内层作用域向外层作用域逐级查找；如果局部作用域未找到，再查找全局常量和全局变量。

变量信息使用 `VariableInfo` 表示：

```cpp
struct VariableInfo {
    bool isConst = false;
    bool hasConstValue = false;
    std::int32_t constValue = 0;
};
```

其中 `isConst` 用于检查常量不可赋值，`hasConstValue` 和 `constValue` 用于编译期常量表达式求值。

### 6.2 全局名称检查

ToyC 要求全局函数、全局变量和全局常量名称唯一。项目使用 `occupiedGlobalNames_` 记录已经出现过的全局名称。如果后续出现同名全局项目，则报告：

```text
redefinition of global name ...
```

该规则可以防止函数和变量重名，也可以防止全局变量之间、函数之间重复定义。

### 6.3 函数定义与调用检查

函数签名由 `FunctionSignature` 表示：

```cpp
struct FunctionSignature {
    ValueType returnType = ValueType::Void;
    std::size_t paramCount = 0;
};
```

语义分析中有两个函数相关结构：

- `result_.program.functions`：记录整个程序中的函数。
- `visibleFunctions_`：记录当前位置之前已经定义的函数。

当分析到一个函数定义时，会先将当前函数加入 `visibleFunctions_`，再分析函数体。这样当前函数内部可以递归调用自身；但如果调用一个定义在后面的函数，则在 `visibleFunctions_` 中找不到，会报告：

```text
call to function f before its definition or to an undeclared function
```

这正好符合 ToyC 中“函数调用必须写在被调函数声明之后，支持递归”的约束。

函数调用还会检查实参数量是否与形参数量一致。如果不一致，则报告类似：

```text
function f expects 2 arguments
```

### 6.4 `main` 函数检查

语义分析结束前会检查程序是否包含入口函数 `main`。合法入口必须满足：

```c
int main()
```

如果缺少 `main`，报告：

```text
missing entry function main
```

如果 `main` 返回类型不是 `int` 或带有参数，报告：

```text
main must have type int main()
```

### 6.5 局部声明与作用域遮蔽

局部变量、局部常量和形参都记录在当前作用域中。同一作用域内不能重复声明同名变量，例如：

```c
int a = 1;
int a = 2;
```

会报告局部重定义错误。但内层作用域可以遮蔽外层作用域：

```c
int a = 1;
{
    int a = 2;
}
```

这是通过作用域栈自然实现的。进入块语句时压入新作用域，离开块语句时弹出作用域。

### 6.6 常量表达式检查

ToyC 要求常量初始化表达式必须在编译期可确定。项目中实现了 `evaluateConstExpr()`，用于对表达式进行编译期求值。

支持的常量表达式包括：

- 整数字面量
- 已声明的常量
- 常量之间的一元运算
- 常量之间的二元算术、关系和逻辑运算

对于逻辑与、逻辑或，常量求值也实现了短路语义。例如：

```c
const int a = 0 && (1 / 0);
```

左侧已经为假，因此右侧不需要求值。这样可以保持与 C 风格逻辑运算一致的行为。

如果常量初始化表达式依赖变量、函数调用或其他无法编译期确定的值，则报告：

```text
const initializer must be a compile-time constant expression
```

全局变量和全局常量的初始化也要求为编译期常量，否则报告：

```text
global initializer must be a compile-time constant expression
```

### 6.7 控制流语义检查

语义分析器使用 `loopDepth_` 记录当前是否处于循环内部。进入 `while` 时递增，离开时递减。分析 `break` 和 `continue` 时，如果 `loopDepth_ <= 0`，则说明它们出现在循环外部，需要报告错误。

对于返回语句，语义分析器根据当前函数返回类型检查：

- `int` 函数必须 `return expr;`
- `void` 函数可以 `return;`
- `void` 函数不能 `return expr;`

此外，`int` 函数需要在所有可能路径上返回值。当前实现中的 `stmtAlwaysReturns()` 支持对 `return`、块语句和 `if/else` 的保守判断。只有当 `if` 的 then 分支和 else 分支都必定返回时，整个 `if/else` 才被认为必定返回。

例如下面程序会被拒绝：

```c
int f(int x) {
    if (x) return 1;
}
```

因为当 `x == 0` 时没有返回值。

## 七、RISC-V32 代码生成

代码生成由 `src/codegen.cpp` 实现，入口函数为：

```cpp
void generateRiscV(
    const CompUnit& program,
    const ProgramInfo& info,
    const CompileOptions& options,
    std::ostream& output
);
```

代码生成器遍历语义分析后的 AST，结合 `ProgramInfo` 中的全局信息，输出 RISC-V32 汇编。

### 7.1 数据段生成

全局变量会生成到 `.data` 段中：

```asm
    .data
    .globl a
a:
    .word 1
```

全局常量不会生成独立存储空间，而是在引用时直接作为立即数加载。这减少了不必要的内存访问。

### 7.2 函数代码结构

每个函数生成如下基本结构：

```asm
    .globl function_name
    .type function_name, @function
function_name:
    addi sp, sp, -frame_size
    sw ra, offset(sp)
    sw s0, offset(sp)
    addi s0, sp, frame_size
    ...
exit_label:
    lw ra, offset(sp)
    lw s0, offset(sp)
    addi sp, sp, frame_size
    ret
    .size function_name, .-function_name
```

函数开头建立栈帧，保存返回地址 `ra` 和帧指针 `s0`；函数结束统一恢复现场并返回。所有 `return` 语句都会跳转到统一的退出标签，避免重复生成函数尾声。

### 7.3 栈帧布局

代码生成器会在生成函数前扫描函数体，为形参和局部声明分配栈槽。栈帧大小按 16 字节对齐。

局部变量和形参绑定信息使用 `Binding` 表示：

```cpp
struct Binding {
    enum class Kind {
        LocalVar,
        LocalConst,
        GlobalVar,
        GlobalConst,
    };

    Kind kind;
    int offset;
    std::int32_t constValue;
};
```

这种设计使代码生成阶段能够统一处理局部变量、局部常量、全局变量和全局常量。

### 7.4 表达式生成

表达式结果统一放入 `a0`。例如：

- 整数字面量：`li a0, value`
- 局部变量：`lw a0, offset(s0)`
- 全局变量：先 `la` 加载地址，再 `lw` 读取值
- 常量：直接 `li a0, constValue`

二元表达式采用简单栈式策略：

1. 生成左表达式，结果在 `a0`。
2. 将 `a0` 压栈。
3. 生成右表达式，结果在 `a0`。
4. 弹出左值到 `t0`。
5. 使用 `t0` 和 `a0` 生成最终结果。

例如加法表达式会生成：

```asm
    add a0, t0, a0
```

关系运算使用 `slt`、`seqz`、`snez` 等指令生成 0 或 1。

### 7.5 短路逻辑生成

逻辑与 `&&` 和逻辑或 `||` 需要短路求值，不能简单作为普通二元运算处理。

对于 `a && b`：

1. 先生成 `a`。
2. 如果 `a == 0`，直接跳转到 false 标签。
3. 否则生成 `b`，并将其规范化为 0 或 1。

对于 `a || b`：

1. 先生成 `a`。
2. 如果 `a != 0`，直接跳转到 true 标签。
3. 否则生成 `b`，并将其规范化为 0 或 1。

测试用例 `tests/milestone7/short_circuit.tc` 中包含：

```c
int main() {
    int a = 0;
    if (a && (1 / a)) {
        return 1;
    }
    if (1 || (1 / a)) {
        return 0;
    }
    return 2;
}
```

如果短路逻辑实现错误，该程序会执行除零表达式；当前实现通过跳转结构避免了这一问题。

### 7.6 控制流生成

`if/else` 语句通过标签和条件跳转实现：

```asm
    beqz a0, else_label
    ...
    j end_label
else_label:
    ...
end_label:
```

`while` 语句生成循环头标签、循环体标签和结束标签：

```asm
while_head:
    ...
    beqz a0, while_end
while_body:
    ...
    j while_head
while_end:
```

`break` 和 `continue` 通过循环标签栈实现。进入循环时压入当前循环的继续标签和结束标签；离开循环时弹出。这样嵌套循环中的 `break` 和 `continue` 能够正确跳转到最近一层循环。

### 7.7 函数调用生成

ToyC 函数参数均为 `int`。当前实现按照常见 RISC-V 调用习惯处理前 8 个参数：

- 第 1 到第 8 个参数使用 `a0` 到 `a7`。
- 返回值使用 `a0`。

生成函数调用时，代码生成器先从右到左计算实参并压栈，再将前 8 个参数从栈中弹入参数寄存器，最后生成：

```asm
    call function_name
```

项目中的 `tests/milestone5/eight_params.tc` 验证了 8 参数调用：

```c
int sum8(int a, int b, int c, int d, int e, int f, int g, int h) {
    return a + b + c + d + e + f + g + h;
}

int main() {
    return sum8(1, 2, 3, 4, 5, 6, 7, 8);
}
```

该测试期望返回值为 36。

## 八、`-opt` 与基础优化

编译器支持命令行参数：

```bash
compiler -opt
```

`main.cpp` 会将该参数记录到 `CompileOptions::enableOpt` 中。当前优化主要集中在常量相关处理。

### 8.1 编译期常量折叠

语义分析阶段实现了常量表达式求值。对于 `const` 声明，初始化表达式如果可以在编译期求值，就会记录常量值。

例如：

```c
int main() {
    const int a = 1 + 2 * 3;
    const int b = (a + 4) * 2;
    return b;
}
```

在 `-opt` 模式下，生成的核心代码可以直接加载最终结果：

```asm
    li a0, 22
```

这说明常量声明没有再产生运行时计算。

### 8.2 局部常量存储优化

对于局部常量，如果启用 `-opt`，代码生成器不会为其生成不必要的栈写入。后续引用该常量时，直接加载立即数即可。

未启用优化时，局部常量仍可能被写入栈槽，这样便于保持未优化模式下的行为直观；启用优化后则减少冗余指令。

### 8.3 后续优化空间

当前优化仍然较基础，主要服务于正确性和简单性能提升。后续可以进一步加入：

- 普通表达式常量折叠
- 常量传播
- 死代码删除
- 冗余跳转删除
- 基本块合并
- 更合理的寄存器分配
- 基于 IR 的中端优化框架

## 十、典型问题与解决方案

### 10.1 表达式优先级处理

表达式优先级如果仅依赖单一表达式规则，很容易产生移进规约冲突，也难以直观看出 AST 结构。项目采用多层非终结符方式表达优先级，使加减乘除、关系运算和逻辑运算自然分层。这种方案虽然语法规则较长，但可读性和可维护性较好。

### 10.2 悬空 `else`

`if (cond) if (cond2) stmt; else stmt;` 中的 `else` 应该与最近的 `if` 结合。项目通过 Bison 的优先级声明和 `%prec` 解决该问题，避免产生语法冲突。

### 10.3 定义后调用与递归

ToyC 要求函数只能调用已经定义的函数，但允许函数内部递归调用自己。项目在分析函数定义时先将当前函数加入可见函数表，再分析函数体。这样既允许递归，又能禁止调用后面才定义的函数。

### 10.4 常量表达式与短路

常量表达式求值不仅要处理算术运算，也要处理逻辑短路。如果 `0 && expr` 中的 `expr` 被强行求值，可能错误地触发除零或未声明引用等问题。项目在常量求值和代码生成两个阶段都专门处理 `&&` 和 `||`，保持语义一致。

### 10.5 返回路径检查

`int` 函数必须保证所有执行路径都有返回值。项目采用保守的语法结构分析：普通 `return` 一定返回；块语句中只要出现必定返回的语句，则块后续路径不可达；`if/else` 只有两个分支都必定返回时才整体必定返回。该方法实现简单，能够覆盖课程测试中的主要场景。

### 10.6 后端临时值保存

表达式求值时需要保存左操作数，再计算右操作数。项目采用压栈方式保存临时值，实现简单且不依赖复杂寄存器分配。缺点是会产生较多栈访问，性能不如寄存器分配优化后的代码。考虑到课程实践首先要求正确性，该方案是合理的初始后端设计。

## 十一、不足与改进方向

虽然项目已经形成了从 ToyC 源码到 RISC-V32 汇编的完整编译流程，但仍有一些不足。

### 11.1 缺少独立 IR

当前代码生成直接遍历 AST。这样实现简单，但 AST 更偏向源语言结构，不适合进行通用优化。后续可以引入三地址码形式的 IR，将前端和后端解耦。

可设计的 IR 包括：

- 模块
- 全局对象
- 函数
- 基本块
- 三地址码指令
- 条件跳转和无条件跳转
- 函数调用和返回

有了 IR 后，可以更容易实现控制流分析、数据流分析和中端优化。

### 11.2 优化能力有限

当前 `-opt` 主要处理局部常量和部分常量折叠。对于普通变量表达式、跨语句常量传播、死代码删除等优化尚未实现。后续可以按如下顺序增强：

1. 在 AST 或 IR 层统一折叠常量表达式。
2. 建立基本块，删除不可达代码。
3. 实现局部常量传播。
4. 删除无用赋值和无用表达式。
5. 消除冗余跳转。

### 11.3 寄存器分配简单

当前表达式求值大量使用栈保存临时值，容易产生冗余 `sw` 和 `lw` 指令。后续可以实现简单寄存器池，将临时值优先放入 `t0` 到 `t6` 等临时寄存器中。进一步还可以实现活跃变量分析和图着色寄存器分配。

### 11.4 错误定位信息不完整

AST 中已经设计了 `SourceLocation`，但语法构建阶段还没有充分将 token 的行列信息写入节点。因此当前语义错误信息主要能说明错误类型，定位精度还有提升空间。后续可以使用 Bison location 机制，为每个 AST 节点记录准确源码位置。

### 11.5 调用约定仍可完善

当前实现已经支持前 8 个参数通过 `a0` 到 `a7` 传递，并对额外参数预留了栈传参逻辑。但完整 ABI 还包含调用者保存寄存器、被调用者保存寄存器、栈对齐等更多细节。后续如果要支持更复杂程序，需要进一步完善调用约定实现。

## 十二、实验总结

本次实践完成了一个 ToyC 到 RISC-V32 汇编的编译器原型。项目从词法分析和语法分析开始，逐步构建 AST，再通过语义分析检查作用域、常量、函数、控制流和返回路径等约束，最后生成可组装执行的 RISC-V32 汇编。

在前端实现中，Flex 和 Bison 能够较好地承担词法和语法分析任务。通过表达式分层规则，可以清晰地实现运算符优先级；通过 AST 节点层次，可以准确表示 ToyC 程序结构。

在语义分析中，作用域栈和函数可见表是两个关键结构。作用域栈解决了局部变量声明、遮蔽和查找问题；函数可见表则自然实现了“定义后调用”和“允许递归”的语言规则。常量表达式求值同时服务于语义检查和后端优化，是连接前端和后端的重要信息。

在后端实现中，项目采用直接遍历 AST 的方式生成 RISC-V32 汇编。该方案实现成本低，便于快速形成完整闭环。通过统一函数出口、栈帧布局、标签生成和短路逻辑跳转，编译器能够覆盖 ToyC 的主要控制结构和表达式语义。

测试方面，项目按 milestone 构建了从基础表达式到复杂语义错误的分层测试集，并提供自动化脚本完成编译、组装、链接、运行和返回值检查。这样的测试结构有助于在扩展功能时及时发现回归问题。

总体来看，本项目已经实现了 ToyC 编译器的核心功能，具备较完整的前端检查和基础 RISC-V32 后端能力。后续如果继续改进，可以重点引入 IR、加强优化管线、完善寄存器分配和错误定位，从而进一步提升工程质量和生成代码性能。
